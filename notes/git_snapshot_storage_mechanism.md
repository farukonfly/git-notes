# 💾 Git 存储机制详解：快照 vs 空间占用

这是一个非常经典的问题！虽然 Git 在**逻辑上**把每一次提交都看作是一个完整的**快照 (Snapshot)**，但在**物理存储**层面，它使用了极其聪明的优化手段，使得仓库体积非常小。

## 1. 并没有真的保存很多“副本”

### ✅ 对于没有修改的文件
如果文件在两个版本之间没有发生变化，Git **绝对不会**再次存储该文件。
*   Git 会计算文件的哈希值 (SHA-1)。
*   如果文件内容相同，Git 仅仅是创建一个指向已有文件对象（Blob）的**指针**。
*   **结果**：无论你提交多少次，只要文件没改，它在磁盘上只占一份空间。

### ✅ 对于被修改的文件
如果一个 10MB 的文件你只改了一行代码，Git 确实会暂时存一个新的 10MB 对象（称为 Loose Object）。
**但是！** Git 会定期通过 **垃圾回收 (GC)** 机制将些松散对象打包成 **Packfiles**。

## 2. 秘密武器：Packfiles (打包文件)

Git 会在特定时机（比如 `git push` 时，或者手动运行 `git gc` 时）寻找内容相似的文件。

*   **增量存储 (Delta Compression)**：Git 会存储文件的**一个**完整版本，然后对于其他相似的版本，只存储它们之间的**差异 (Delta)**。
*   这就像 SVN 等增量备份系统做的那样，但 Git 是在底层存储时自动处理的，对用户透明。

## 3. Zlib 压缩

所有存储在 `.git/objects` 目录下的对象（不管是文件内容、提交信息还是目录结构），在写入磁盘之前都会经过 **Zlib 压缩**。文本文件的压缩率非常高。

## 🧪 总结

| 层面 | 行为描述 |
| :--- | :--- |
| **逻辑层面 (用户看到的)** | 每次 Commit 都是完整的项目快照，像时刻对整个文件夹拍照。 |
| **物理层面 (磁盘存储)** | 1. **去重**：相同文件只存一份。<br>2. **压缩**：Zlib 强力压缩。<br>3. **增量**：相似文件只存差异 (Packfiles)。 |

这就是为什么 Git 仓库通常比同等内容的 SVN 仓库还要小，因为它不仅做增量，还拥有全局去重能力（比如把文件 A 复制成文件 B，Git 只需要存一份数据）。
